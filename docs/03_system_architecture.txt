SYSTEM ARCHITECTURE
===================
Project: AI BA Agent
Version: 1.0 (MVP Phase 1)
Date: 2026-03-14

=== ARCHITECTURE OVERVIEW ===
Type: Microservices + Serverless (hybrid)
Pattern: Backend-for-Frontend (BFF) + RAG-enabled AI service
Gateway: Required (single-port entry point)
Deployment: Docker Compose (multi-container)
Port Strategy: Multi-port (separate service ports, gateway routes externally)

=== SERVICE RESPONSIBILITIES ===

**1. BACKEND API SERVICE (Port 5000)**
Language: Python 3.11
Framework: FastAPI
Responsibilities:
- Receive document uploads (recording, email, document)
- Coordinate STT via ElevenLabs Scribe v2 API
- Call CrewAI agents for data extraction + summarization
- Manage document versioning & storage
- Expose REST endpoints for frontend
- Audit logging for all operations

Key Endpoints:
  POST   /api/documents/upload        Upload recording / document
  POST   /api/documents/process        Trigger processing pipeline
  GET    /api/documents/{id}           Retrieve document + versions
  GET    /api/documents/{id}/versions  List all versions
  POST   /api/approval/submit          Submit for approval
  GET    /api/approval/status          Check approval status
  POST   /api/audit/logs               Query audit trail

**2. AUTH SERVICE (Port 5001)**
Language: Python 3.11
Framework: FastAPI + JWT
Responsibilities:
- User login/logout
- Token generation & refresh
- Role-based access control (RBAC)
- Project-level permission checks
- Two-factor authentication (optional Phase 2)

Key Endpoints:
  POST   /auth/login                   Authenticate user
  POST   /auth/refresh                 Renew token
  GET    /auth/me                      Current user info
  GET    /auth/permissions             User's permissions by project

**3. RAG SERVICE (Port 5002)**
Language: Python 3.11
Framework: FastAPI + LangChain
Responsibilities:
- Manage RAG knowledge base in Qdrant
- Index documents into vector DB
- Similarity search across project scope
- Answer user questions with source citations
- Maintain project-scoped memory boundaries

Key Endpoints:
  POST   /rag/index                    Add document to KB
  GET    /rag/search                   Semantic search
  POST   /rag/chat                     Query chatbot
  GET    /rag/sources/{doc_id}         Retrieve source evidence

**4. GATEWAY (Port 80/443)**
Type: Nginx Reverse Proxy
Responsibilities:
- Single entry point for all client requests
- Route traffic to appropriate backend service
- SSL/TLS termination
- Rate limiting & DDoS protection
- Request logging & access control
- Static frontend asset serving

Routes:
  / → Frontend (static)
  /api/* → Backend API (5000)
  /auth/* → Auth Service (5001)
  /rag/* → RAG Service (5002)
  /health → Health check endpoint

**5. n8n WORKFLOW ENGINE (Port 8080)**
Type: No-code workflow automation
Responsibilities:
- Listen for Gmail triggers
- Poll Google Sheets for backlog status
- Execute daily/weekly cron jobs
- Dispatch Telegram notifications
- Handle ElevenLabs webhook callbacks (STT complete)
- Manage inter-service orchestration

Workflows:
  - Email ingestion trigger
  - Backlog scanning & notification
  - Weekly report generation
  - STT webhook receiver

**6. DATABASE LAYER**

PostgreSQL (Port 5432)
- Primary relational database
- Tables:
  * users (user_id, email, role, created_at, updated_at)
  * projects (project_id, name, owner, created_at, access_level)
  * documents (doc_id, project_id, title, type, upload_time, status)
  * document_versions (version_id, doc_id, version_num, content_hash, created_by, approval_status)
  * audit_logs (log_id, action, user_id, doc_id, timestamp, details)
  * approval_workflows (workflow_id, doc_id, status, approvers, current_step)
  * user_permissions (user_id, project_id, role, granted_by, granted_at)

Qdrant Vector DB (Port 6333)
- Purpose: RAG knowledge base
- Collections:
  * documents_embeddings (project_id: partition key, doc_id, title, content_vector, metadata)
  * query_cache (recent user queries, response pairs)
- Search: Cosine similarity, filtered by project_id
- Replication: Optional (Phase 2 for HA)

Redis (Port 6379) - OPTIONAL
- Session storage
- Rate limit tracking
- Real-time notification pub/sub

=== CONTAINERIZATION STRATEGY ===

All major services are containerized. No container-in-container required.

Services to containerize:
✓ Frontend (Node.js build → Nginx serving static)
✓ Backend API (Python FastAPI)
✓ Auth Service (Python FastAPI)
✓ RAG Service (Python FastAPI + LangChain)
✓ Gateway (Nginx reverse proxy)
✓ PostgreSQL (official PostgreSQL image)
✓ Qdrant (official Qdrant image)
✓ Redis (official Redis image, optional)

Services NOT containerized (external):
- n8n (if using SaaS n8n.io, otherwise containerize)
- ElevenLabs (external API)
- Google Drive / Sheets / Gmail (external APIs)
- OpenRouter / DeepSeek (external APIs)
- Telegram (external API)

=== DEPLOYMENT SEQUENCE ===

1. **Database Layer** (PostgreSQL, Qdrant, Redis)
   - Must be deployed first
   - Run migrations to initialize schema
   - Other services depend on database connectivity

2. **Backend Services** (Auth, Backend API, RAG)
   - Depends on: Database layer
   - Can be deployed in parallel
   - Requires database connection string, API keys

3. **n8n Workflow Engine** (if self-hosted)
   - Depends on: Backend services (for webhooks)
   - Configure n8n workflows after backend is accessible

4. **Frontend** (React + Nginx)
   - Depends on: Backend API reachable
   - Build optimized static assets
   - Configure API endpoint URLs

5. **Gateway** (Nginx)
   - Last to deploy
   - Route all traffic after all services are healthy
   - TLS certificates must be in place

=== SINGLE-PORT vs MULTI-PORT DECISION ===

**Decision: MULTI-PORT (with Gateway)**

Justification:
- Services are independent (clarity of responsibility)
- Easier local development (each dev runs services individually)
- Easier scaling (deploy specific service without restarting all)
- Easier debugging (check each service health independently)
- Backend services need internal communication (5000 ↔ 5002 for RAG calls)

Gateway handles:
- Single external entry point (Port 80/443)
- Internal routing to backend services
- TLS termination
- Rate limiting
- Access logging

=== GATEWAY ARCHITECTURE ===

Technology: Nginx (reverse proxy)

Configuration:
```yaml
# Nginx runs in container on Port 80/443
# Routes:
#   / → Frontend static files
#   /api/* → Backend API (5000)
#   /auth/* → Auth Service (5001)
#   /rag/* → RAG Service (5002)
#   /health → 200 OK (liveness probe)

upstream backend {
  server backend:5000;
}

upstream auth {
  server auth:5001;
}

upstream rag {
  server rag:5002;
}

server {
  listen 80;
  listen 443 ssl;
  
  # TLS config
  ssl_certificate /etc/nginx/certs/cert.pem;
  ssl_certificate_key /etc/nginx/certs/key.pem;
  
  # Redirect HTTP to HTTPS
  if ($scheme != "https") {
    return 301 https://$server_name$request_uri;
  }
  
  # Frontend (static)
  location / {
    root /usr/share/nginx/html;
    try_files $uri /index.html;
  }
  
  # Backend API
  location /api/ {
    proxy_pass http://backend;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }
  
  # Auth Service
  location /auth/ {
    proxy_pass http://auth;
  }
  
  # RAG Service
  location /rag/ {
    proxy_pass http://rag;
    # WebSocket support for real-time chat
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
  }
  
  # Health check
  location /health {
    return 200 "OK";
  }
}
```

=== SECURITY ARCHITECTURE ===

**Authentication Flow:**
1. User login at /auth/login (username + password)
2. Auth Service validates, returns JWT token
3. Frontend stores token in secure httpOnly cookie
4. All API requests include Authorization: Bearer {token}
5. Backend validates token signature & expiry
6. Backend checks user role & project permissions

**Authorization (RBAC):**
- Admin: Full access to all projects, user management
- BA / PM / Tech Lead: Standard user, can view/edit own project documents
- Business Owner / Legal: Can approve documents in assigned projects
- IT: Read-only access across all projects

**Data Protection:**
- PII masking per role (e.g., phone numbers → ****-****-**)
- Encrypted storage for sensitive fields (contracts, financial data)
- Audit logging: Every action logged with user, timestamp, details
- Access logs: All API requests logged with user, IP, endpoint
- Retention: Audit logs retained for 12 months minimum
