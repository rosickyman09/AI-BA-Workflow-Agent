Stage 3 — Requirement Analysis
Purpose
Take docs/01_project_requirement.txt and instruct Copilot to deeply analyze and restructure it into a comprehensive docs/01_requirement_analysis.md — covering business goals, system modules, risks, and deployment assumptions — before any architecture or code is produced.

Prerequisites

 docs/01_project_requirement.txt exists and is complete (Stage 2 done)
 .copilot-instructions.md is in place at project root
 docs/ folder exists at project root


Step 1 — Open Copilot Chat and Attach the Input File

Open Copilot Chat (Ctrl+Shift+I / Cmd+Shift+I)
Attach or reference the input file:

   docs/01_project_requirement.txt

Use the full prompt in Step 2 below


Step 2 — Copilot Prompt
Copy and paste the following prompt into Copilot Chat:
Here is my project requirement.

Please analyze and structure it into:

1. Business goal
2. Functional modules
3. Non-functional requirements
4. Data requirements
5. AI capabilities required
6. Risks, assumptions, and dependencies
7. Deployment constraints
8. Containerization and environment assumptions

Then convert them into system modules and create:
docs/01_requirement_analysis.md

Step 3 — Expected Output Structure
Copilot should produce docs/01_requirement_analysis.md with the following sections:
markdown# Requirement Analysis

## 1. Business Goal
- Core problem being solved
- Target users and their needs
- Success metrics

## 2. Functional Modules
- Module name + responsibility
- Inputs and outputs per module
- Dependencies between modules

## 3. Non-Functional Requirements
- Performance targets
- Security requirements
- Scalability needs
- Availability / uptime expectations

## 4. Data Requirements
- Data entities and relationships
- Storage needs (type, volume, retention)
- Data flow between modules

## 5. AI Capabilities Required
- Where AI/Copilot is used in the system
- Type of AI capability needed (generation, classification, search, etc.)
- Data or context AI needs to function

## 6. Risks, Assumptions, and Dependencies
- Known risks and mitigation strategies
- Assumptions being made about users, data, or environment
- External dependencies (APIs, services, teams)

## 7. Deployment Constraints
- Environment targets (local, cloud, on-prem)
- CI/CD requirements
- Release and rollback constraints

## 8. Containerization and Environment Assumptions 🐳
- Which services will be containerized
- Base images and environment assumptions
- Single-port vs multi-port decision
- Gateway requirement (yes/no and reason)
- Docker Compose as orchestration method (confirm or note alternative)
- Environment variable and secrets strategy

Docker Checklist for Section 8 🐳

Output
